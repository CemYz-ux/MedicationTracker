# User Flows — MedicationTracker

Living source of truth for what the site does. Update this file with every feature and bug fix.

## Add a medication
**Status:** Implemented
**Flow:**
1. A circular floating action button (FAB), showing only a bare "+" glyph (accessible name "Add medication" via `aria-label`, mirroring the close control's "×" + `aria-label="Close"` pattern), is fixed to the bottom-right corner of the viewport at all times — on every viewport size, whether the list is empty or full, and regardless of scroll position (MED-23). It is the only control that opens the add-medication form. Its 56x56px touch target is deliberately larger than the app's ~44px baseline (MED-19), being the app's single primary action. The medication list reserves bottom space so the FAB never overlaps the last row of cards.
2. Activating the control (click, or Enter/Space via keyboard) opens a modal dialog (native `<dialog>` + `showModal()`) containing Name, Dose, and Interval (hours) fields.
3. User fills in Name, Dose, and Interval (hours) and submits.
4. If Name, Dose, or Interval is empty, or Interval is non-numeric or not greater than zero, an inline error message is shown inside the modal (`role="alert"`), the modal stays open, and nothing is saved.
5. On a valid submit, the medication is added with `lastTakenAt: null` (Active — logging/cooldown are owned by "Log and cancel a dose (tap the card)" below and not part of this flow), the modal closes, and the medication appears in the medication list.
6. The medication and its Name/Dose/Interval persist across page reloads (stored in `localStorage`).
7. The modal can be closed at any time — via its close control, the Cancel button, pressing Escape, or clicking the backdrop — without saving. Every close path discards unsaved input, resets the form, and returns keyboard focus to the FAB. The close control's own touch target is ~44x44px (MED-19), the app's baseline that the FAB itself exceeds (see above).
8. While the modal is open, Tab-based keyboard focus is contained inside it and cannot reach the page behind it.

## View medications
**Status:** Implemented
**Flow:**
1. There is no static page title/subtitle chrome or storage disclaimer footer — the page opens straight into the grey list card, whose heading shows today's day and date (e.g. "Sunday, July 12"), computed client-side from the browser's clock once on page load (not a live-updating clock).
2. On page load, previously logged medications are read from `localStorage` and rendered in a list.
3. Each row is a single tappable card (see "Log and cancel a dose (tap the card)" below), compact and short (MED-33): a top strip holds the (cooldown-only) countdown text on the left and the Edit (pencil-icon), Delete ("x"), and Reset ("↻") controls grouped together on the right (see "Edit a medication", "Delete a medication", and "Reset a cooldown" below); below that, a second row holds the medication's name and dose (stacked, name bold above dose in smaller grey text) — with an Active-only "last taken" relative-time line stacked beneath them (MED-38, see "Last taken" below) — on the left, and the always-visible interval stat — a large numeral plus a small "HOURS" caption (see "Interval readout" below) — on the right.
4. There is no status-pill text or dot anywhere on the card (MED-33 removed it) — state is communicated purely by the card's background tint and border color: a light green tint/border when the card is tappable-to-log (Active), or an amber tint/border while a logged dose's interval is still counting down (Cooldown). There is no third "Paused" state (MED-32 introduced one; MED-34 removed it again) — a card is always either Active or Cooldown, decided fresh each time from the same underlying "is this medication in cooldown" check the countdown text and Reset use (see "Log and cancel a dose (tap the card)").
5. If there are no medications (including corrupted/missing stored data), an empty-state message ("No medications yet — add one to get started.") is shown instead of the list.
6. Records left over from the old prototype schema (`{id, name, dose, time}`, no `intervalHours`) are silently discarded on load rather than shown or migrated.
7. The list lays out as a responsive CSS Grid so more medications are visible at once on wider screens (MED-22): a single column below 640px viewport width (pixel-identical to the original single-column layout), 2 columns from 640px up to 999px, and 3 columns from 1000px up. The page's content-width cap grows at those same two breakpoints so no column ever renders narrower than about 16rem, while still staying capped (not stretching edge-to-edge) on ultra-wide monitors. A list shorter than the current column count fills the leftmost cells only and never stretches to fill the empty ones, and a card's own fixed height (see "Cooldown countdown and fill") holds across its row-sharing siblings too — one card changing status never jitters the height of the row it shares with others. The grid uses plain DOM-order auto-placement (no CSS `order`), so visual left-to-right/top-to-bottom reading order always matches keyboard tab order.

## Log and cancel a dose (tap the card)
**Status:** Implemented
**Flow (MED-32 replaced the old separate GO and Stop buttons with whole-card tap-to-toggle; neither button renders anywhere anymore. MED-32 also introduced a third "Paused" state — tap-to-freeze, tap-again-to-resume-from-that-frozen-point — which MED-34 removed again after real-world use showed it wasn't the behavior wanted. The flow below is the current, two-state behavior.):**
1. Each medication's entire card body (the top strip's countdown text and the name/dose/interval-stat row below it — everything except the Edit/Delete/Reset icon buttons in the top strip) is one tappable region: a click, or Enter/Space while it holds keyboard focus, activates it. It is keyboard-reachable (Tab) and exposed as a `button`-role control with an accessible name that identifies both the medication and what the *next* tap will do: "Log {name} dose now" while Active, or "Cancel {name} cooldown" while a cooldown is running.
2. Tapping an **Active** card logs a dose: records the current time as `lastTakenAt`, snapshots the interval that is active at that moment (so a later edit to the interval can't retroactively change this cooldown — see "Cooldown countdown and fill"), and saves both to `localStorage`. The card immediately shows Cooldown (amber tint/border) at 100% fill.
3. Tapping a **Cooldown** card cancels it immediately and reverts the medication to Active — the same outcome the pre-MED-32 Stop button (and, briefly, MED-32's "Revert to Active" Edit-dialog control) produced, just triggered by tapping the card itself: `lastTakenAt` and the cooldown's interval snapshot are cleared, the card returns to the green Active tint/border with no countdown/fill, and the change is saved to `localStorage` immediately. There is no intermediate "Paused"/frozen state to pass through — a single tap on a Cooldown card cancels it outright.
4. A medication's cooldown also still reverts to Active entirely on its own, with no user action, once its interval naturally elapses (see "Cooldown countdown and fill") — unaffected by this flow.
5. A screen-reader-only status message is announced via the page-level `aria-live="polite"` region for each of these actions: "{name} logged." or "{name} reverted to Active.". Keyboard focus stays on the card itself after either action (never teleported elsewhere), mirroring the app's general "never leave focus on/near a control that just changed meaning without saying so" discipline.
6. If saving fails (e.g. storage full/unavailable), an inline error ("Could not update — please try again.") is shown on that card, the medication's displayed state and in-memory data are left exactly as they were, and nothing is written to storage — the UI never implies success that wasn't persisted.
7. None of these actions on one medication's card ever affect any other medication's card or data.
8. The Edit, Delete, and Reset icon buttons living in the same card's top strip are independently keyboard-reachable and clickable and never trigger the card's own tap-to-toggle action (they're siblings of the tappable region in the markup, not nested inside it).
9. **Backward compatibility:** a medication record saved to `localStorage` before MED-34 shipped may still carry a leftover, no-longer-read `pausedRemainingMs` field from the old Paused state. Nothing in the current code reads that field anymore (there is no migration step that strips it either) — it is simply inert and can never resurrect paused-looking behavior on that card.

## Last taken
**Status:** Implemented
**Flow (MED-38):**
1. Every **Active** card shows a small "last taken" relative-time line beneath the name/dose (e.g. "Just now", "12m ago", "3h 5m ago", "1d 2h ago"), derived from `lastTakenAt` — or the literal "Not yet taken" when the medication has never been logged. This line updates live, ticking forward on the same periodic re-check that already drives the Cooldown countdown (see "Cooldown countdown and fill"), so it never goes stale while a card stays Active.
2. **Cooldown** cards never show this line at all — no new line, no layout change there. The countdown text already visible during Cooldown implies a recent take, so this is deliberately Active-only. This means an Active card is now slightly taller than a Cooldown card for the same medication (a change from the prior MED-18/MED-33 "identical height in both states" behavior — Cooldown's own height is unaffected).
3. When a cooldown elapses on its own and a card reactivates (see "Cooldown countdown and fill"), this line is correct and live from the very first render after flipping to Active — never a stale or "0s ago"/"Just now" flash, since it always reflects the real elapsed time since `lastTakenAt`, not time since reactivation.
4. Cancelling a Cooldown by tapping the card (see "Log and cancel a dose (tap the card)") clears `lastTakenAt` back to `null`, so the reverted-to-Active card shows "Not yet taken" again, the same as a never-logged medication.

## Interval readout
**Status:** Implemented
**Flow (MED-34 introduced this as plain text; MED-33 restyled it as the compact numeral+caption stat described below):**
1. Every medication card — Active or Cooldown alike — shows a compact, always-visible stat of how often it's taken: a large numeral (the medication's `intervalHours`, e.g. "8" or "4.5") with a small uppercase "HOURS" caption beneath it, on the right side of the name/dose row. Unlike the Cooldown countdown's "{remaining} left" text (see "Cooldown countdown and fill"), this stat is visible in *every* state, not just during Cooldown.
2. The stat always reflects the medication's live, currently-set `intervalHours` — not the value a running cooldown snapshotted when it started (`cooldownIntervalHours`, used only by the countdown's own total). Editing the interval via the Edit dialog updates this stat immediately, even mid-cooldown, without disturbing the running countdown/fill itself (see "Edit a medication").
3. This is now the full MED-33 compact-card treatment (numeral + caption), not the earlier MED-34 plain-text stopgap ("Every 8h") — see "View medications" for how it sits alongside the rest of the redesigned card.

## Reset a cooldown
**Status:** Implemented
**Flow (MED-32):**
1. Every medication card shows a small "↻" Reset control in its top strip (alongside Edit and Delete), labeled for assistive tech as "Reset {medication name} cooldown", reachable/operable via keyboard alone, and — unlike the old Stop button it partly supersedes — always visible and usable regardless of the card's current state (Active or Cooldown).
2. Activating Reset (from *either* state) immediately starts a brand-new, full-length cooldown as of that instant — the same effect as tapping an Active card.
3. Reset is saved to `localStorage` immediately, keyboard focus stays on the Reset button, and a screen-reader-only status message ("{medication name} cooldown reset.") is announced.
4. If saving fails, an inline error ("Could not reset — please try again.") is shown on that card and nothing about the medication's displayed or stored state changes.
5. Resetting one medication never affects any other medication's data or displayed state.

## Edit a medication
**Status:** Implemented
**Flow:**
1. Every medication card (Active or Cooldown) shows a small pencil-icon "Edit" control in its top strip, labeled for assistive tech as "Edit {medication name}" and reachable/operable via keyboard alone (Tab to focus, Enter/Space to activate) — sitting alongside the Delete ("x") and Reset ("↻") controls. All three controls' touch targets are ~44x44px (MED-19) — the glyph itself stays visually small; only the clickable area around it is sized up.
2. Activating a card's Edit control opens a modal dialog (native `<dialog>` + `showModal()`) containing Name, Dose, and Interval (hours) fields, pre-filled with that specific medication's current saved values — never blank, and never another medication's values. This is a separate dialog element from the Add-medication modal (mirrors its markup/CSS classes and interaction pattern, but is not a shared/mode-toggling dialog). Interval's control lives here now (MED-32) — the old per-card inline Interval input is gone.
3. User changes Name, Dose, and/or Interval, and submits.
4. If Name or Dose is empty, or Interval is non-numeric or not greater than zero, an inline error message is shown inside the modal (`role="alert"`), the modal stays open, and nothing at all is saved (not even the fields that *were* valid) — mirroring the Add-medication modal's own validation.
5. On a valid submit, that medication's `name`, `dose`, and `intervalHours` are updated (Name/Dose trimmed) — `lastTakenAt` and the cooldown interval snapshot are left completely untouched, so editing never disturbs a running cooldown's countdown, status, or card fill. An Active medication stays Active; editing never starts a cooldown. The change is saved to `localStorage` immediately, the modal closes, and the card reflects the new values right away, including the always-visible interval stat (see "Interval readout"). A live-running cooldown continues to use the interval that was active when it started (see "Cooldown countdown and fill") — an edited Interval only governs the *next* fresh cooldown (the next tap-on-Active or Reset).
6. If saving fails (e.g. storage full/unavailable), an inline error ("Could not save changes — please try again.") is shown inside the modal, the modal stays open, and the card keeps showing the pre-edit values — the UI never implies a change was saved when it wasn't persisted.
7. The modal can be closed at any time — via its close control, the Cancel button, pressing Escape, or clicking the backdrop — without saving. Every close path discards unsaved input and returns keyboard focus to the *specific* card's Edit control that opened it (not a generic default), including after a successful save, even though saving re-renders the entire list. The close control's own touch target is ~44x44px (MED-19), the same baseline as the Add-medication modal's close control (see above).
8. While the modal is open, Tab-based keyboard focus is contained inside it and cannot reach the page behind it (native `<dialog>` modal behavior, same mechanism the Add-medication modal relies on).
9. The edited Name/Dose/Interval persist across a page reload.
10. Editing one medication's Name/Dose/Interval never affects any other medication's data (name, dose, interval, cooldown state) or display.
11. **No "Revert to Active" control here anymore:** MED-32 briefly added a "Revert to Active" button to this dialog as the replacement for the removed Stop button. MED-34 removed that dialog control again, since tapping a Cooldown card now does exactly the same job (cancel and revert to Active — see "Log and cancel a dose (tap the card)") without needing to open this dialog at all.

## Delete a medication
**Status:** Implemented
**Flow:**
1. Every medication card (Active or Cooldown) shows a small "x" Delete control in its top strip, next to the Edit and Reset controls, labeled for assistive tech as "Delete {medication name}" and reachable/operable via keyboard alone (Tab to focus, Enter/Space to activate) — reuses the same `.icon-btn` base styling as Edit and Reset, with its own hover treatment.
2. Activating the control (click, or Enter/Space via keyboard) deletes that medication immediately — no confirmation dialog or intermediate step. This is a deliberate product decision: a medication's definition is trivial to re-add, and the only real cost of an accidental delete is losing an in-progress cooldown, judged an acceptable tradeoff for this app's scope.
3. Deletion is permanent. There is no undo.
4. The medication is removed from `localStorage` immediately, so it does not reappear after a page reload.
5. Deleting a medication that is currently in Cooldown discards its countdown/timestamp data the same way as deleting an Active one — there is no special-casing by status.
6. Deleting one medication never affects any other medication's data or displayed state (Active/Cooldown status, countdown, values) — same multi-medication independence principle established elsewhere in this app.
7. Deleting the last remaining medication returns the list to the empty state defined in "View medications" ("No medications yet — add one to get started."). The Add-medication FAB remains visible and usable, since it lives outside the medication list.
8. Keyboard focus moves to a nearby row's Delete control immediately after a delete succeeds, since the deleted row's own controls (including the Delete control that had focus) no longer exist in the DOM: the row that shifted up into the deleted row's position, or the new last row if the deleted row was last. Only when the list is now completely empty does focus move to the Add-medication FAB instead — the correct target once there is no row left at all.
9. If saving the deletion fails (e.g. storage full/unavailable), an inline error ("Could not delete — please try again.") is shown next to that row, the medication is not removed from the displayed list, and nothing is written to storage — the UI never implies a deletion that wasn't persisted, mirroring this app's other storage-failure disciplines.

## Cooldown countdown and fill
**Status:** Implemented
**Flow (MED-33 relocated the countdown text from the card's bottom-right corner to the top-left of the top strip and shortened its wording; the underlying cooldown timing/fill logic below is unchanged):**
1. A medication is "in cooldown" whenever `now < lastTakenAt + (the interval that was active when it was last logged)` — this is a live time comparison, not a one-way flag, so once that moment passes the medication automatically becomes Active again. There is no exception to this: MED-32 briefly introduced one (a **Paused** medication was always treated as in cooldown regardless of elapsed time), and MED-34 removed both the exception and the Paused state it existed for.
2. While in Cooldown, the card has an amber tint/border (no status-pill text — see "View medications") and shows countdown text in the shorter form "{remaining} left" (e.g. "3h 12m 45s left"), at the top-left of the card's top strip (MED-33) — sharing that strip with the Edit/Delete/Reset icon row on the right (see "Edit a medication"). That strip's space is permanently reserved on every card (Active or Cooldown) so showing/hiding the countdown never grows or shrinks the card — no layout shift in the list when a card changes status (MED-18, extended by MED-33). The countdown counts down to the exact second live (MED-29). As with the pre-MED-29 hours/minutes format, a zero-value component is omitted rather than shown as "0" — down to the seconds level too (e.g. "45s left" under a minute left, "12m 45s left" under an hour left). This text is distinct from the separate, always-visible interval stat (see "Interval readout"), which reflects the medication's live `intervalHours` rather than a cooldown snapshot.
3. The card's background fill starts at 100% the instant a dose is logged and recedes from right to left in direct proportion to elapsed time while Cooldown is running — e.g. at the halfway point of an 8-hour interval, exactly the left half of the card is coloured.
4. A page-wide periodic check re-evaluates every medication's cooldown status roughly once a second (MED-29), so the countdown text — including its live seconds component — and fill stay current, and a Cooldown medication flips back to Active (fill reaching exactly 0% at the same moment) with no reload or user interaction required once its interval elapses. Each medication's refresh is independent — one medication's cooldown state or timing never affects another's. The re-check only ever touches the specific card(s) it re-evaluates (countdown text, fill, and the tap-target's accessible name), never re-rendering the whole list, which keeps the faster cadence cheap.
5. If the Interval is edited (via the Edit dialog — see "Edit a medication") while a medication is mid-cooldown, the countdown and fill continue to reflect the interval that was active when the cooldown started, not the newly edited value — the edited value only governs the *next* fresh cooldown (a later tap-on-Active or Reset). The always-visible interval stat (see "Interval readout") updates immediately regardless, since it deliberately tracks the live value instead.
6. An Active (not-in-cooldown) card shows no fill or countdown text at all — both are Cooldown-only. (The interval stat is the exception: it is visible in both states — see "Interval readout".)
