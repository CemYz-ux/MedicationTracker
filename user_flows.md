# User Flows — MedicationTracker

Living source of truth for what the site does. Update this file with every feature and bug fix.

## Add a medication
**Status:** Implemented
**Flow:**
1. A circular floating action button (FAB), showing only a bare "+" glyph (accessible name "Add medication" via `aria-label`, mirroring the close control's "×" + `aria-label="Close"` pattern), is fixed to the bottom-right corner of the viewport at all times — on every viewport size, whether the list is empty or full, and regardless of scroll position (MED-23). It is the only control that opens the add-medication form. Its 56x56px touch target is deliberately larger than the app's ~44px baseline (MED-19), being the app's single primary action. The medication list reserves bottom space so the FAB never overlaps the last row of cards.
2. Activating the control (click, or Enter/Space via keyboard) opens a modal dialog (native `<dialog>` + `showModal()`) containing Name, Dose, and Interval (hours) fields.
3. User fills in Name, Dose, and Interval (hours) and submits.
4. If Name, Dose, or Interval is empty, or Interval is non-numeric or not greater than zero, an inline error message is shown inside the modal (`role="alert"`), the modal stays open, and nothing is saved.
5. On a valid submit, the medication is added with `lastTakenAt: null` and `pausedRemainingMs: null` (Active — logging/cooldown/pausing are owned by "Log, pause, and resume a dose (tap the card)" below and not part of this flow), the modal closes, and the medication appears in the medication list.
6. The medication and its Name/Dose/Interval persist across page reloads (stored in `localStorage`).
7. The modal can be closed at any time — via its close control, the Cancel button, pressing Escape, or clicking the backdrop — without saving. Every close path discards unsaved input, resets the form, and returns keyboard focus to the FAB. The close control's own touch target is ~44x44px (MED-19), the app's baseline that the FAB itself exceeds (see above).
8. While the modal is open, Tab-based keyboard focus is contained inside it and cannot reach the page behind it.

## View medications
**Status:** Implemented
**Flow:**
1. There is no static page title/subtitle chrome or storage disclaimer footer — the page opens straight into the grey list card, whose heading shows today's day and date (e.g. "Sunday, July 12"), computed client-side from the browser's clock once on page load (not a live-updating clock).
2. On page load, previously logged medications are read from `localStorage` and rendered in a list.
3. Each row is a single tappable card (see "Log, pause, and resume a dose (tap the card)" below) showing the medication's name, dose, a status pill, and (cooldown/paused-only) live countdown text (see "Cooldown countdown and fill"). A small Edit (pencil-icon), Delete ("x"), and Reset ("↻") control are grouped together in the card's header (see "Edit a medication", "Delete a medication", and "Reset a cooldown" below).
4. The status pill is a green "Active" pill when the card is tappable-to-log, an amber "Cooldown" pill while a logged dose's interval is still counting down, or an amber "Paused" pill while that countdown is frozen (MED-32). Cooldown and Paused both reflect the same underlying "is this medication in cooldown" check the countdown text and Reset use — no separate logic of its own; Paused only differs in that its countdown is additionally frozen (see "Log, pause, and resume a dose (tap the card)").
5. If there are no medications (including corrupted/missing stored data), an empty-state message ("No medications yet — add one to get started.") is shown instead of the list.
6. Records left over from the old prototype schema (`{id, name, dose, time}`, no `intervalHours`) are silently discarded on load rather than shown or migrated.
7. The list lays out as a responsive CSS Grid so more medications are visible at once on wider screens (MED-22): a single column below 640px viewport width (pixel-identical to the original single-column layout), 2 columns from 640px up to 999px, and 3 columns from 1000px up. The page's content-width cap grows at those same two breakpoints so no column ever renders narrower than about 16rem, while still staying capped (not stretching edge-to-edge) on ultra-wide monitors. A list shorter than the current column count fills the leftmost cells only and never stretches to fill the empty ones, and a card's own fixed height (see "Cooldown countdown and fill") holds across its row-sharing siblings too — one card changing status never jitters the height of the row it shares with others. The grid uses plain DOM-order auto-placement (no CSS `order`), so visual left-to-right/top-to-bottom reading order always matches keyboard tab order.

## Log, pause, and resume a dose (tap the card)
**Status:** Implemented
**Flow (MED-32 — replaces the old separate GO and Stop buttons entirely; neither renders anywhere anymore):**
1. Each medication's entire card body (name/dose, status pill, and countdown — everything except the Edit/Delete/Reset icon buttons in its header) is one tappable region: a click, or Enter/Space while it holds keyboard focus, activates it. It is keyboard-reachable (Tab) and exposed as a `button`-role control with an accessible name that identifies both the medication and what the *next* tap will do: "Log {name} dose now" while Active, "Pause {name} cooldown" while a cooldown is running, or "Resume {name} cooldown" while paused.
2. Tapping an **Active** card logs a dose: records the current time as `lastTakenAt`, snapshots the interval that is active at that moment (so a later edit to the interval can't retroactively change this cooldown — see "Cooldown countdown and fill"), and saves both to `localStorage`. The card immediately shows Cooldown at 100% fill.
3. Tapping a **running (unpaused) Cooldown** card pauses it: the remaining time is frozen exactly where it stood at that instant, and the card's pill changes from "Cooldown" to "Paused". Pausing does **not** revert the medication to Active and does **not** reset it — it is purely a freeze. The frozen state is saved to `localStorage` immediately and survives a reload/reopen.
4. Tapping a **Paused** card resumes it: the countdown continues from exactly the frozen remaining value (not restarted, and not recomputed against whatever real time actually passed while paused) — e.g. a card paused with 3h12m45s left still has exactly 3h12m45s left the instant it's resumed, no matter how long it sat paused.
5. A **Paused** medication never reactivates on its own, no matter how much time passes while it's paused — the usual "cooldown ends automatically once its interval elapses" behavior (see "Cooldown countdown and fill") is suspended for the entire time a medication is paused, and only resumes counting down once explicitly resumed.
6. A screen-reader-only status message is announced via the page-level `aria-live="polite"` region for each of these actions: "{name} logged.", "{name} cooldown paused.", or "{name} cooldown resumed.". Keyboard focus stays on the card itself after any of these actions (never teleported elsewhere), mirroring the app's general "never leave focus on/near a control that just changed meaning without saying so" discipline.
7. If saving fails (e.g. storage full/unavailable), an inline error ("Could not update — please try again.") is shown on that card, the medication's displayed state and in-memory data are left exactly as they were, and nothing is written to storage — the UI never implies success that wasn't persisted.
8. None of these actions on one medication's card ever affect any other medication's card or data.
9. The Edit, Delete, and Reset icon buttons living in the same card's header are independently keyboard-reachable and clickable and never trigger the card's own tap-to-toggle action (they're siblings of the tappable region in the markup, not nested inside it).

## Reset a cooldown
**Status:** Implemented
**Flow (MED-32):**
1. Every medication card shows a small "↻" Reset control in its header (alongside Edit and Delete), labeled for assistive tech as "Reset {medication name} cooldown", reachable/operable via keyboard alone, and — unlike the old Stop button it partly supersedes — always visible and usable regardless of the card's current state (Active, Cooldown, or Paused).
2. Activating Reset (from *any* state) immediately starts a brand-new, full-length cooldown as of that instant — the same effect as tapping an Active card, and if the medication was Paused, its paused/frozen data is fully cleared (not left stale) so the new cooldown behaves like any freshly-logged one.
3. Reset is saved to `localStorage` immediately, keyboard focus stays on the Reset button, and a screen-reader-only status message ("{medication name} cooldown reset.") is announced.
4. If saving fails, an inline error ("Could not reset — please try again.") is shown on that card and nothing about the medication's displayed or stored state changes.
5. Resetting one medication never affects any other medication's data or displayed state.

## Edit a medication
**Status:** Implemented
**Flow:**
1. Every medication card (Active, Cooldown, or Paused) shows a small pencil-icon "Edit" control in its header, labeled for assistive tech as "Edit {medication name}" and reachable/operable via keyboard alone (Tab to focus, Enter/Space to activate) — sitting alongside the Delete ("x") and Reset ("↻") controls. All three controls' touch targets are ~44x44px (MED-19) — the glyph itself stays visually small; only the clickable area around it is sized up.
2. Activating a card's Edit control opens a modal dialog (native `<dialog>` + `showModal()`) containing Name, Dose, and Interval (hours) fields, pre-filled with that specific medication's current saved values — never blank, and never another medication's values. This is a separate dialog element from the Add-medication modal (mirrors its markup/CSS classes and interaction pattern, but is not a shared/mode-toggling dialog). Interval's control lives here now (MED-32) — the old per-card inline Interval input is gone.
3. User changes Name, Dose, and/or Interval, and submits.
4. If Name or Dose is empty, or Interval is non-numeric or not greater than zero, an inline error message is shown inside the modal (`role="alert"`), the modal stays open, and nothing at all is saved (not even the fields that *were* valid) — mirroring the Add-medication modal's own validation.
5. On a valid submit, that medication's `name`, `dose`, and `intervalHours` are updated (Name/Dose trimmed) — `lastTakenAt`, the cooldown interval snapshot, and (MED-32) any paused/frozen remaining time are all left completely untouched, so editing never disturbs a running or paused cooldown's countdown, status, or card fill. An Active medication stays Active; editing never starts a cooldown. The change is saved to `localStorage` immediately, the modal closes, and the card reflects the new values right away. A live-running cooldown continues to use the interval that was active when it started (see "Cooldown countdown and fill") — an edited Interval only governs the *next* fresh cooldown (the next tap-on-Active, Reset, or a paused card's resume).
6. If saving fails (e.g. storage full/unavailable), an inline error ("Could not save changes — please try again.") is shown inside the modal, the modal stays open, and the card keeps showing the pre-edit values — the UI never implies a change was saved when it wasn't persisted.
7. The modal can be closed at any time — via its close control, the Cancel button, pressing Escape, or clicking the backdrop — without saving. Every close path discards unsaved input and returns keyboard focus to the *specific* card's Edit control that opened it (not a generic default), including after a successful save, even though saving re-renders the entire list. The close control's own touch target is ~44x44px (MED-19), the same baseline as the Add-medication modal's close control (see above).
8. While the modal is open, Tab-based keyboard focus is contained inside it and cannot reach the page behind it (native `<dialog>` modal behavior, same mechanism the Add-medication modal relies on).
9. The edited Name/Dose/Interval persist across a page reload.
10. Editing one medication's Name/Dose/Interval never affects any other medication's data (name, dose, interval, cooldown state) or display.
11. **Revert to Active** (MED-32, the resolved replacement for the removed Stop button): a "Revert to Active" button inside this same dialog, shown only while the medication being edited is in Cooldown or Paused (hidden entirely while Active, since there's nothing to revert). Activating it immediately cancels the cooldown — from either a running Cooldown or a Paused state — returns the medication to Active exactly as if its interval had simply elapsed naturally (green pill, no countdown/fill, tap-target reads "Log ... dose now" again), saves the change, closes the dialog, returns keyboard focus to that row's Edit control, and announces "{medication name} reverted to Active." via the live status region. It is a plain button (not the form's submit control), so it never triggers Name/Dose/Interval validation. If saving fails, an inline error ("Could not revert to Active — please try again.") is shown inside the (still-open) dialog and the cooldown is left running/paused exactly as it was.

## Delete a medication
**Status:** Implemented
**Flow:**
1. Every medication card (Active, Cooldown, or Paused) shows a small "x" Delete control in its header, next to the Edit and Reset controls, labeled for assistive tech as "Delete {medication name}" and reachable/operable via keyboard alone (Tab to focus, Enter/Space to activate) — reuses the same `.icon-btn` base styling as Edit and Reset, with its own hover treatment.
2. Activating the control (click, or Enter/Space via keyboard) deletes that medication immediately — no confirmation dialog or intermediate step. This is a deliberate product decision: a medication's definition is trivial to re-add, and the only real cost of an accidental delete is losing an in-progress cooldown, judged an acceptable tradeoff for this app's scope.
3. Deletion is permanent. There is no undo.
4. The medication is removed from `localStorage` immediately, so it does not reappear after a page reload.
5. Deleting a medication that is currently in Cooldown or Paused discards its countdown/timestamp/paused data the same way as deleting an Active one — there is no special-casing by status.
6. Deleting one medication never affects any other medication's data or displayed state (Active/Cooldown/Paused status, countdown, values) — same multi-medication independence principle established elsewhere in this app.
7. Deleting the last remaining medication returns the list to the empty state defined in "View medications" ("No medications yet — add one to get started."). The Add-medication FAB remains visible and usable, since it lives outside the medication list.
8. Keyboard focus moves to a nearby row's Delete control immediately after a delete succeeds, since the deleted row's own controls (including the Delete control that had focus) no longer exist in the DOM: the row that shifted up into the deleted row's position, or the new last row if the deleted row was last. Only when the list is now completely empty does focus move to the Add-medication FAB instead — the correct target once there is no row left at all.
9. If saving the deletion fails (e.g. storage full/unavailable), an inline error ("Could not delete — please try again.") is shown next to that row, the medication is not removed from the displayed list, and nothing is written to storage — the UI never implies a deletion that wasn't persisted, mirroring this app's other storage-failure disciplines.

## Cooldown countdown and fill
**Status:** Implemented
**Flow:**
1. A medication is "in cooldown" whenever `now < lastTakenAt + (the interval that was active when it was last logged)` — this is a live time comparison, not a one-way flag, so once that moment passes the medication automatically becomes Active again. The one deliberate exception (MED-32) is a **Paused** medication: it is always treated as in cooldown, for as long as it stays paused, regardless of how much time passes — see "Log, pause, and resume a dose (tap the card)".
2. While in Cooldown or Paused, the card is visually dimmed (amber "Cooldown"/"Paused" pill, tinted border — Paused additionally shown with a dashed border) and shows countdown text in the form "{remaining} of {total} remaining" (e.g. "3h 12m 45s of 5h remaining"), pinned to the card's bottom-right corner. That corner's space is permanently reserved on every card (Active, Cooldown, or Paused) so showing/hiding the countdown never grows or shrinks the card — no layout shift in the list when a card changes status (MED-18). While Cooldown is running, the *remaining* portion counts down to the exact second live (MED-29); while Paused, it is frozen at exactly the value it held the instant the card was paused, and does not move again until resumed. The *total* portion always stays a static hours/minutes label (e.g. "5h"), since it reflects the interval a cooldown started with, not something that changes live even while running. As with the pre-MED-29 hours/minutes format, a zero-value component is omitted rather than shown as "0" — down to the seconds level too (e.g. "45s of 5h remaining" under a minute left, "12m 45s of 5h remaining" under an hour left).
3. The card's background fill starts at 100% the instant a dose is logged and recedes from right to left in direct proportion to elapsed time while Cooldown is running — e.g. at the halfway point of an 8-hour interval, exactly the left half of the card is coloured. While Paused, the fill is frozen at exactly the fraction it held at the moment of pausing.
4. A page-wide periodic check re-evaluates every medication's cooldown status roughly once a second (MED-29), so the countdown text — including its live seconds component — and fill stay current, and a *running* (unpaused) Cooldown medication flips back to Active (fill reaching exactly 0% at the same moment) with no reload or user interaction required once its interval elapses. A **Paused** medication is exempt from this reactivation for as long as it stays paused, no matter how much real time this periodic check observes passing. Each medication's refresh is independent — one medication's cooldown/paused state or timing never affects another's. The re-check only ever touches the specific card(s) it re-evaluates (countdown text, fill, and the tap-target's accessible name), never re-rendering the whole list, which keeps the faster cadence cheap.
5. If the Interval is edited (via the Edit dialog — see "Edit a medication") while a medication is mid-cooldown or paused, the countdown and fill continue to reflect the interval that was active when the cooldown started, not the newly edited value — the edited value only governs the *next* fresh cooldown (a later tap-on-Active, Reset, or a paused card's resume).
6. An Active (not-in-cooldown) card shows no fill or countdown text at all — both are Cooldown/Paused-only.
