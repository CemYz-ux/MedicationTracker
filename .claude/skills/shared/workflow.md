## Handoff Requirements

### product-owner ‚Üí solution-architect
The PBI must have:
- User Story in *As a / I want / So that* format
- At least 3 AC (incl. one error/edge case)
- UX description or wireframe for any UI change
- PO self-review checklist passed (`user_story_review.md`)

### solution-architect ‚Üí product-owner (feedback round)
The Architect returns:
- Feasibility verdict (‚úÖ / ‚öÝÔ∏è / ‚ùå)
- List of required changes to AC, UX, or scope
- Architecture notes for the Developer (to be captured in arc42)
- Any decision the PO must make before development starts

### product-owner ‚Üí frontend-developer
The Developer must receive:
- The improved PBI with all Architect feedback incorporated
- The arc42 section generated by the Architect
- No open questions on either document

### frontend-developer ‚Üí code-reviewer
The PR must have:
- All AC self-verified locally by the Developer
- Vitest unit tests passing for all new pure logic
- Playwright E2E test for the primary flow
- `user_flows.md` and `README.md` updated
- Branch name and PR title following conventions

### code-reviewer ‚Üí frontend-developer (if changes needed)
The reviewer returns:
- All blocking comments labeled `blocking:`
- Specific line references and clear remediation instructions

### frontend-developer ‚Üí qa-acceptance-tester
After PR approval and merge:
- PBI link
- Environment (GitHub Pages deploy or local preview build)
- Any test data / localStorage seeding notes

### qa-acceptance-tester ‚Üí product-owner (if all AC pass)
- Brief test summary attached to PBI
- PBI moved to `Ready for PO Review`

### qa-acceptance-tester ‚Üí frontend-developer (if bugs found)
- One Bug work item per failure with: steps to reproduce, expected, actual, evidence, severity
- All bugs linked to the original PBI
- PBI returned to `Active`

### product-owner ‚Üí default branch merge approval
After PO acceptance:
- PBI closed
- Signal to **frontend-developer** to merge feature branch to the default branch

## Jira Workflow Integration

Every handoff in the chain above must be reflected in Jira. See `shared/jira.md` for connection details and MCP call syntax. Use `getTransitionsForJiraIssue` to get the exact transition IDs before calling `transitionJiraIssue`.

| Workflow event | Jira action |
|----------------|-------------|
| PO creates PBI | `createJiraIssue` (Story, status = **To Do**) |
| PO sends PBI to Architect | `addCommentToJiraIssue` ‚Äî note sent for arch review |
| Architect returns feedback | `addCommentToJiraIssue` on the Story with feedback |
| PO incorporates feedback, PBI cleared | `addCommentToJiraIssue` ‚Äî "PBI cleared by architect, ready for dev gate" |
| PBI passes Ready-for-Dev gate | `addCommentToJiraIssue` ‚Äî "Passed all 7 gates. Handed to developer." |
| Developer picks up PBI | `transitionJiraIssue` ‚Üí **In Progress** |
| Developer opens PR | `addCommentToJiraIssue` ‚Äî include PR link |
| PR merged, handed to QA | `addCommentToJiraIssue` ‚Äî "PR merged. Handed to QA for acceptance testing." |
| QA approves all AC | `addCommentToJiraIssue` ‚Äî attach test summary; `transitionJiraIssue` ‚Üí **Done** (pending PO review) |
| QA finds bugs | `createJiraIssue` per bug (Bug type) + `createIssueLink` to Story; `addCommentToJiraIssue` on Story with bug refs |
| Bug fixed | `addCommentToJiraIssue` on Bug ‚Äî "Fixed in <commit/PR>. Notifying QA for re-test." |
| Bug verified | `transitionJiraIssue` on Bug ‚Üí **Done** |
| PO final acceptance | `addCommentToJiraIssue` on Story ‚Äî "PO accepted. Approved for merge to the default branch." |
| Feature merged to the default branch | `transitionJiraIssue` on Story ‚Üí **Done** (if not already) |
